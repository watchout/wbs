## 1. プロダクト概要

### 1.1 一言で何か（エレベーターピッチ）

```
弱電・電気工事会社（10〜100名規模） 向けの 現場業務DXプラットフォーム で、
ホワイトボード・Excel文化による情報共有の非効率 を解決する。
紙やホワイトボードでの予定管理 とは異なり、
リアルタイムサイネージ + Googleカレンダー連携 + AIスケジュール調整 を提供する。
```

### 1.2 背景・課題

現状の課題を具体的に記述:

- **ホワイトボード依存**: 事務所のホワイトボードに手書きで週間予定を管理。外出先から確認不可。更新漏れが日常的に発生
- **情報の非対称**: 事務所にいる事務員と現場作業員の間で予定情報に時差がある。朝礼でしか全体共有ができない
- **Excelの限界**: 一部デジタル化している企業もExcelで管理。同時編集不可、バージョン管理不可、モバイル表示が困難
- **デジタルツールの壁**: 既存のプロジェクト管理ツール（Asana, Backlog等）は建設現場の作業員にとって複雑すぎる
- **多言語対応不足**: 外国人技能実習生が増加しているが、日本語のみのツールが多い

### 1.3 解決策

本プロダクトがどのように課題を解決するか:

- **リアルタイムサイネージ**: 事務所のモニターに週間予定を常時表示。Socket.IO による1秒以下の即時反映
- **マルチデバイス**: PC・タブレット・大型モニターで同一データを閲覧。PWA対応でオフラインでも24時間キャッシュ
- **Googleカレンダー双方向連携**: 既存のカレンダーから予定を自動同期。二重入力を完全排除
- **AI日程調整**: 参加者の空き状況をAIが分析し、最適な会議スケジュールを提案
- **多言語対応**: ja, en, vi, zhHans, fil, ne, ptBR の7言語に対応
- **ゼロ学習コスト**: ホワイトボードの見た目をデジタルで再現。操作はブラウザを開くだけ

## 4. スコープ定義

### 4.1 Goals（やること - Phase 0 MVP）

- [x] **[MUST]** 週間スケジュールボード（CRUD + リアルタイム同期）
- [x] **[MUST]** サイネージ表示（大画面対応、自動更新）
- [x] **[MUST]** ユーザー管理（CRUD, ロール: ADMIN/LEADER/MEMBER/DEVICE）
- [x] **[MUST]** 部門管理（CRUD、ソフトデリート）
- [x] **[MUST]** メール/パスワード認証 + デバイスログイン
- [x] **[MUST]** Googleカレンダー双方向同期（OAuth2, Webhook, 増分同期）
- [x] **[SHOULD]** AI日程調整（候補スロット生成、回答収集、確定）
- [ ] **[SHOULD]** 多言語対応（7言語）
- [ ] **[SHOULD]** オフラインキオスク（PWA + IndexedDB 24hキャッシュ）

### 4.2 Non-Goals（やらないこと - Phase 0）

明示的にスコープ外とするもの:

- 在庫・資材管理（STOCK - Phase 2）
- 車両管理・アルコールチェック（DRIVE - Phase 3）
- 日報・工数管理（LOG - Phase 4）
- AIコンシェルジュ高度機能（自然言語による操作 - Phase 2以降）
- ネイティブモバイルアプリ（PWAで対応）
- SSO / SAML 連携
- 請求・決済機能

### 4.3 将来の検討事項（Future Considerations）

今回は対象外だが、将来検討するもの:

- マルチモジュール統合（WEEK + STOCK + DRIVE）
- AI高度分析（工数集計、原価計算）
- 外部システム連携API（公開API）
- ミエルファイル（変更検知・履歴管理機能 - M&A価値向上目的）
- ホワイトラベル対応

## 6. 制約条件

### 6.1 技術制約

- Node.js 20 + Nuxt 3 + Prisma ORM（PostgreSQL）
- Prisma ORM のみ使用（`$queryRaw` / `$executeRaw` 禁止）
- マルチテナント: 全データに `organizationId` スコープ必須
- Socket.IO 4.x でリアルタイム通信
- Docker + Docker Compose でデプロイ
- GitHub Actions CI/CD

### 6.2 ビジネス制約

| 制約種別 | 内容 |
|---------|------|
| 納期 | Phase 0 MVP: 2026年Q1中 |
| 予算 | 初期投資 ¥0（開発は AI駆動、インフラは ConoHa VPS 月額 ¥1,000〜3,000 程度） |
| 人員 | 1名 + AI（Claude Code / Cursor） |
| 法規制 | 個人情報保護法準拠（メール、名前の取り扱い） |

### 6.3 運用制約

- ConoHa VPS（OpenStack）でのホスティング
- Nginx リバースプロキシ + Let's Encrypt SSL
- PostgreSQL WAL をオブジェクトストレージにバックアップ
- Terraform による IaC 管理

## §3-E 入出力例

> PRDレベルのユースケース入出力例。各UCの正常系2ケース＋異常系3ケースを定義。

### UC-001: 週間スケジュール表示・編集

| # | 種別 | 入力 | 期待出力 |
|---|------|------|----------|
| 1 | 正常 | LEADER がブラウザから月曜の予定を新規作成（件名: 現場A、時間: 9:00-17:00） | 予定がリアルタイムで全端末に反映（<1秒）、サイネージにも即時表示 |
| 2 | 正常 | ADMIN が既存予定を編集（件名変更 + 時間変更） | 変更が保存され、Socket.IO経由で全クライアントに即時通知 |
| 3 | 異常 | 未認証ユーザーがAPI直接アクセス（/api/schedules） | 401 Unauthorized、ログイン画面へリダイレクト |
| 4 | 異常 | MEMBER が他テナントの予定を取得試行（organizationId改竄） | 403 Forbidden、自テナントデータのみ返却 |
| 5 | 異常 | 件名を空で送信 | 400 Bad Request、バリデーションエラー「件名は必須です」 |

### UC-003: Googleカレンダー同期

| # | 種別 | 入力 | 期待出力 |
|---|------|------|----------|
| 1 | 正常 | ADMIN がGoogleアカウントをOAuth2で連携開始 | Google認可画面→コールバック→トークン保存→初回フル同期実行 |
| 2 | 正常 | Google側で予定追加→Webhook受信 | 増分同期で新しい予定がWBS側に反映 |
| 3 | 異常 | Googleトークンが期限切れ | リフレッシュトークンで自動更新、失敗時はユーザーに再認証を促す |
| 4 | 異常 | Webhook署名が不正 | 400 Bad Request、同期処理は実行しない |
| 5 | 異常 | Google API のレート制限超過 | 指数バックオフで再試行、3回失敗でエラーログ＋Sentry通知 |

## §3-F 境界値

> PRDレベルの主要データ項目の境界パターン。

| データ項目 | 最小値 | 最大値 | 境界値テスト |
|-----------|--------|--------|-------------|
| 予定件名（title） | 1文字 | 200文字 | 空文字→エラー、1文字→OK、200文字→OK、201文字→エラー |
| 予定日（date） | 当日-1年 | 当日+2年 | 範囲外の日付→エラー |
| 週間表示範囲 | 月曜 | 日曜 | 7日間固定、週跨ぎの予定は分割表示 |
| ユーザー数/組織 | 1名 | 500名 | 0名→組織作成不可、500名超→警告表示 |
| 部署数/組織 | 0 | 50 | 50超→エラー「上限に達しました」 |
| パスワード長 | 8文字 | 128文字 | 7文字→エラー、8文字→OK、128文字→OK、129文字→エラー |
| OTPコード | 000000 | 999999 | 6桁未満→バリデーションエラー、7桁以上→バリデーションエラー |
| Socket.IO同時接続 | 1 | 200（推定上限） | 200接続超→新規接続を拒否せず性能低下を許容 |
| Google同期イベント数 | 0 | 2500/カレンダー | 2500超→ページネーションで処理 |

## §3-G 例外応答

> 全ユースケースに対する例外応答定義。

| エラーケース | HTTPステータス | エラーコード | ユーザー向けメッセージ |
|-------------|---------------|-------------|---------------------|
| 未認証アクセス | 401 | UNAUTHORIZED | ログインが必要です |
| 権限不足（ロール不適合） | 403 | FORBIDDEN | この操作を行う権限がありません |
| 他テナントデータアクセス | 403 | TENANT_FORBIDDEN | アクセスが拒否されました |
| リソース未存在 | 404 | NOT_FOUND | 指定されたデータが見つかりません |
| バリデーション失敗 | 400 | VALIDATION_ERROR | 入力内容を確認してください: {details} |
| ログイン失敗（5回） | 423 | ACCOUNT_LOCKED | アカウントがロックされました。15分後に再試行してください |
| OTP検証失敗（3回） | 400 | OTP_MAX_ATTEMPTS | 認証コードの入力回数上限に達しました。再送信してください |
| OTP未検証で課金操作 | 403 | OTP_REQUIRED | セキュリティ認証が必要です |
| Googleトークン無効 | 401 | GOOGLE_TOKEN_EXPIRED | Googleアカウントの再連携が必要です |
| レートリミット超過 | 429 | RATE_LIMITED | リクエストが多すぎます。しばらくお待ちください |
| サーバー内部エラー | 500 | INTERNAL_ERROR | システムエラーが発生しました。しばらくお待ちください |
| DB接続エラー | 503 | SERVICE_UNAVAILABLE | サービスが一時的に利用できません |

## §3-H Gherkin シナリオ

> MUST要件の受入テストシナリオ。

### Scenario: 週間スケジュールの新規作成（UC-001）

```gherkin
Scenario: LEADERが週間スケジュールに新規予定を作成する
  Given LEADERロールのユーザーがログイン済みである
  And 週間スケジュールボード画面を表示している
  When 月曜日の時間枠をクリックする
  And 件名「現場A点検」、時間「09:00-17:00」を入力する
  And 「保存」ボタンをクリックする
  Then 予定がデータベースに保存される
  And 同一組織の全クライアントにSocket.IOで即時通知される
  And サイネージ画面にも1秒以内に反映される
```

### Scenario: マルチテナントデータ分離（SEC）

```gherkin
Scenario: 他テナントのデータにアクセスできないことを検証する
  Given 組織Aに属するADMINユーザーがログイン済みである
  And 組織Bにスケジュールデータが存在する
  When 組織BのスケジュールIDでAPIリクエストを送信する
  Then 403 Forbiddenが返される
  And レスポンスに組織Bのデータは含まれない
```

### Scenario: デバイスログイン（AUTH-004）

```gherkin
Scenario: サイネージ端末がデバイスログインする
  Given 組織のkioskSecretが発行済みである
  When orgSlugとkioskSecretでデバイスログインAPIを呼び出す
  Then DEVICEロールのJWTトークンが発行される
  And サイネージ画面に自動遷移する
```

### Scenario: Googleカレンダー双方向同期（UC-003）

```gherkin
Scenario: Google側の予定追加がWBSに反映される
  Given ユーザーがGoogleカレンダーを連携済みである
  And Webhook URLが登録済みである
  When Google側で新しい予定を追加する
  And GoogleからWebhook通知が送信される
  Then 増分同期が実行される
  And WBSの週間ボードに新しい予定が表示される
```

### Scenario: OTP 2FA による課金操作保護（SEC）

```gherkin
Scenario: ADMINがOTP認証なしで課金操作を試みる
  Given ADMINロールのユーザーがログイン済みである
  And OTP認証が未実施である
  When サブスクリプション開始APIを呼び出す
  Then 403エラーが返される
  And レスポンスに requireOtp: true が含まれる
```
