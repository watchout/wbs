## 1. 変更管理プロセス

### 1.1 フロー

```
┌─────────────┐
│ 変更要求    │ ← 誰でも起票可能
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 影響分析    │ ← TL/担当者が分析
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 審議・承認  │ ← 変更委員会（または承認者）
└──────┬──────┘
       │
   ┌───┴───┐
   │       │
 承認    却下
   │       │
   ▼       ▼
┌─────┐ ┌─────┐
│実装 │ │記録 │
└──┬──┘ └─────┘
   │
   ▼
┌─────────────┐
│ 完了・検証  │
└─────────────┘
```

### 1.2 変更の分類

| 分類 | 説明 | 承認者 | 目安期間 |
|------|------|-------|---------|
| 緊急 | 本番障害対応など | TL即断 | 即時 |
| 大規模 | スコープ・スケジュールに大きな影響 | PO + ステークホルダー | 1週間 |
| 中規模 | 1週間以上の追加作業 | PO | 3日 |
| 小規模 | 軽微な変更 | TL | 1日 |

## 6. 変更管理の原則

### 6.1 原則

1. **全ての変更を記録する**: 口頭での合意も記録に残す
2. **影響分析を必ず行う**: 「簡単な変更」も影響を確認
3. **適切な承認を得る**: 分類に応じた承認者の承認を得る
4. **ドキュメントを更新する**: 変更後はSSOTを更新

### 6.2 やってはいけないこと

- ❌ 承認なしで実装を進める
- ❌ 影響分析をスキップする
- ❌ 変更後にドキュメントを更新しない
- ❌ 変更履歴を残さない
