# AI開発フレームワーク v3.0 - マスターガイド

> 「こんなの作りたい」の一言から、AI開発可能な仕様書一式を生成し、
> 開発を開始するためのフレームワーク

---

## このフレームワークでできること

```
INPUT:  「○○みたいなサービス作りたいんだけど...」

OUTPUT: ・事業設計資料（アイデア検証、ペルソナ、競合、価値提案）
        ・プロダクト設計（PRD、機能仕様、UI設計）
        ・技術設計（API、DB、横断的関心事、開発規約）
        ・マーケティング設計（LP、SNS、メール、ローンチ、価格）
        ・CLAUDE.md が設定済みのプロジェクト
        → Claude Code で即座に開発開始できる状態
```

## 始め方（3つのステップ）

### ステップ1: Claude.ai でディスカバリーを開始

Claude.ai を開き、以下を送信してください:

```
あなたはAI開発のプロダクトマネージャーです。
以下のディスカバリーフローに従って、
私のアイデアをヒアリングしてください。

[08_DISCOVERY_FLOW.md の内容をペースト]

私のアイデア: ○○みたいなサービスを作りたい。
```

AIが Stage 1〜5 の質問を開始します。約30-60分の対話で、アイデアが構造化されます。

### ステップ2: 生成チェーンで仕様書を作成

ディスカバリー完了後、同じ会話（またはClaude.aiのProjectsに資料を格納して）で:

```
ディスカバリーの結果をもとに、
10_GENERATION_CHAIN.md のStep 1に従って
IDEA_CANVAS を作成してください。

[10_GENERATION_CHAIN.md の該当Stepをペースト]
[templates/idea/IDEA_CANVAS.md のテンプレートをペースト]
```

Step 1-A → 1-B → 1-C → 1-D → Step 2 → Step 3 と順番に進めます。

### ステップ3: Claude Code でプロジェクトを構築

Step 3 に到達したら、Claude Code で:

```bash
# プロジェクト初期構築
./setup-project.sh my-project

# AIが仕様書をファイルとして配置
cd my-project
claude "以下の仕様書内容を docs/ 配下に配置して:
       [Step 1-2 で生成した仕様書をペースト]"

# CLAUDE.md を設定
claude "templates/project/CLAUDE.md の
       {{}} を実際の値で置換して"

# 技術スタックに基づいてスキャフォールド
claude "docs/standards/TECH_STACK.md に基づいて
       プロジェクトを初期化して"
```

Claude Code でプロジェクトを開けば、CLAUDE.md が自動読み込みされ、開発開始です。

## 生成チェーン概要

### Step 0: ディスカバリー → Step 1: 事業設計

```
Discovery回答
    ↓
IDEA_CANVAS（90%）
    ├─→ USER_PERSONA（70%）
    ├─→ COMPETITOR_ANALYSIS（60%）
    └─→ VALUE_PROPOSITION（80%）

※ 各ドキュメントは前のドキュメントをインプットに生成
※ 各ドキュメント後にユーザーが確認・修正
※ Go/No-Go判断で先に進むか決定
```

```
Step 1 全資料
    ↓

    ├─→ FEATURE_CATALOG（90%）
    │     共通機能 / 個別機能に切り分け

    │     → 13_SSOT_AUDIT.md で監査（95点合格）
```

## マーケティング統合

このフレームワークは「作っても売れない」を防ぐために、

## SSOT（Single Source of Truth）階層

```
PRD（SSOT-0）: 何を・なぜ作るか
    ↓
機能カタログ（SSOT-1）: 全機能の一覧・優先度
    ↓
機能仕様書: 各機能の詳細仕様
    ↓
    ├── UI/状態遷移（SSOT-2）: 画面と状態の定義
    ├── API契約（SSOT-3）: エンドポイント・型定義
    ├── データモデル（SSOT-4）: テーブル・関連
    └── 横断的関心事（SSOT-5）: 認証・エラー・ログ
```

**原則: 仕様書がない機能は実装しない**

## よくある質問

### Q: 1人で全部やる必要がある？
A: いいえ。AIが大部分を生成します。あなたの役割は「判断」と「確認」です。

### Q: どの程度のプログラミング経験が必要？
A: ディスカバリーで技術レベルを確認し、レベルに応じた技術スタックを推奨します。ノーコード〜プロエンジニアまで対応。

### Q: マーケティングは必須？
A: 任意です。ディスカバリーのQ5-8でスキップできます。ただし「開発中からリード獲得」は強く推奨。

### Q: 所要時間は？
A: アイデアから開発開始まで1-2日。AI生成→ユーザー確認の繰り返し。

### Q: 既にアイデアが固まっている場合は？
A: ディスカバリーをスキップし、Step 1 の IDEA_CANVAS から直接始められます。

## 知識データ（Knowledge Base）

### 位置づけ

```
知識データはSSOTとは別に、プロジェクトのドメイン知識を蓄積する場所。
Discovery や SSOT生成の品質を高めるためのインプットとして機能する。

  docs/knowledge/（知識データ）
         ↓ インプット
  Discovery → SSOT生成 → 実装
         ↑ フィードバック
  インタビュー、競合調査、市場調査

知識データ ≠ SSOT:
  - SSOT = 実装の唯一の根拠（厳密に管理）
  - 知識データ = SSOTを作るための参考情報（柔軟に蓄積）
```

### ディレクトリ構造

```
docs/knowledge/
├── _INDEX.md                 ← 知識データ一覧・使い方
├── market/                   ← 市場知識
│   ├── competitors.md        ← 競合分析
│   ├── trends.md             ← 市場トレンド
│   └── regulations.md        ← 規制・法令
├── domain/                   ← ドメイン知識
│   ├── terminology.md        ← 業界用語
│   ├── best-practices.md     ← ベストプラクティス
│   └── common-features.md    ← 業界標準機能
└── users/                    ← ユーザー知識
    ├── personas.md           ← 詳細ペルソナ
    ├── pain-points.md        ← 課題詳細
    └── interviews.md         ← インタビュー記録
```

### 参照ルール

```
1. Discovery開始前: docs/knowledge/ の存在を確認
   → 存在する → 関連ファイルを読み込んでからヒアリング
   → 存在しない → 通常通りヒアリング

2. 提案時: 知識データの根拠を示す
   「知識データ（competitors.md）によると…」

3. 不足時: 追加を推奨する
   「この情報は知識データにありません。追加を推奨します」

4. 矛盾時: 報告する
   「ユーザー回答と知識データに矛盾があります」
```

## プロジェクトタイプ別プロファイル

### 概要

```
フレームワークは5種類のプロジェクトタイプをサポートする。
タイプごとに必要なSSOT・監査・Discoveryの範囲が異なる。

framework init --type=<type> で初期化時に指定する。
未指定の場合は app（デフォルト）が適用される。
プロファイル定義: templates/profiles/<type>.json
```

### プロジェクトタイプ一覧

| タイプ | 用途 | 例 |
|--------|------|-----|
| `app` | フルスタックWebアプリ | SaaS、業務システム、予約サービス |
| `lp` | ランディングページ | β募集LP、プロダクトLP、キャンペーンLP |
| `hp` | ホームページ/コーポレートサイト | 会社紹介、サービス紹介 |
| `api` | API/バックエンド専用 | REST API、GraphQL、マイクロサービス |
| `cli` | CLIツール | 開発者向けCLI、自動化ツール |

### タイプ別の有効化マトリクス

```
             SSOT                   Discovery   Marketing   監査
─────────────────────────────────────────────────────────────────
app          全部                   Stage 1-5   任意        全部
             PRD, CATALOG, UI,
             API, DATA, CROSS

lp           PRD, UI のみ           Stage 1-3   必須        Code, Visual

hp           PRD, UI のみ           Stage 1-2   任意        Code, Visual

api          PRD, API, DATA         Stage 1-3   なし        Code, Test
             (+CROSS)

cli          PRD, API               Stage 1-3   なし        Code, Test
```

### タイプ別フロー

```
app（フルスタック）:
────────────────────────────────
  Discovery(全Stage)
    → Business → Product → Technical
    → 全SSOT生成
    → 全監査実行
  最も網羅的。デフォルト。

lp（ランディングページ）:
────────────────────────────────
  Discovery(Stage 1-3: 課題・ターゲット・価値)
    → Business(VALUE_PROPOSITION重視)
    → LP_SPEC + SNS_STRATEGY 生成
    → UI_STATE(ページ構成・ワイヤーフレーム)
    → 実装 → Visual監査
  マーケティングドキュメントが必須。
  API/DB設計は不要。

hp（ホームページ）:
────────────────────────────────
  Discovery(Stage 1-2: 概要・構成)
    → PRD(ページ構成・コンテンツ定義)
    → UI_STATE(ページ構成・レスポンシブ)
    → 実装 → Visual監査
  最もシンプル。Discoveryも最小。

api（API/バックエンド）:
────────────────────────────────
  Discovery(Stage 1-3: 課題・ターゲット・機能)
    → Business → Product
    → API_CONTRACT + DATA_MODEL 集中
    → CROSS_CUTTING(認証・エラー・ログ)
    → 実装 → Code監査 + Test監査
  UI関連は全てスキップ。

cli（CLIツール）:
────────────────────────────────
  Discovery(Stage 1-3: 課題・ターゲット・機能)
    → Business → Product
    → API_CONTRACT(コマンド定義に読み替え)
    → 実装 → Code監査 + Test監査
  API_CONTRACTをCLIコマンド・フラグ定義に読み替える。
  DB設計は通常不要。
```

### プロファイルの自動判定

```
ユーザーがタイプを指定しない場合の自動判定:

Q1: 「LP」「ランディングページ」「β募集」というキーワードがあるか？
  → Yes → lp

Q2: 「ホームページ」「コーポレートサイト」「会社紹介」というキーワードがあるか？
  → Yes → hp

Q3: 「API」「バックエンド」「マイクロサービス」というキーワードがあるか？
  → Yes → api

Q4: 「CLI」「コマンドライン」「ツール」というキーワードがあるか？
  → Yes → cli

Q5: 上記に該当しない
  → app（デフォルト）

判定後、ユーザーに確認:
「プロジェクトタイプを {type} と判定しました。よろしいですか？」
```
