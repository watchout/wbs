## Stage 0: 知識データの事前確認（Discovery開始前）

### 目的
既存の知識データを読み込み、ヒアリングの質を高める

### 手順

```
Step 1: docs/knowledge/ の存在を確認
────────────────────────────────
  存在しない → Stage 1 に直接進む
  存在する → Step 2 へ

Step 2: 関連ファイルを読み込む
────────────────────────────────
  以下の順序で確認:
  1. users/pain-points.md    → 課題の事前把握
  2. users/personas.md       → ペルソナの事前把握
  3. market/competitors.md   → 競合状況の把握
  4. domain/terminology.md   → 業界用語の把握
  5. domain/common-features.md → 業界標準機能の把握

Step 3: 知識サマリーを作成（内部用）
────────────────────────────────
  読み込んだ知識から以下を整理:
  - 既に把握している課題
  - 既に把握しているペルソナ像
  - 競合の状況
  - 業界特有の用語・概念

Step 4: ヒアリング方針を調整
────────────────────────────────
  知識データがある場合:
  - 既知の情報は確認質問に変える
    「○○という課題があると理解していますが、合っていますか？」
  - 未知の部分を重点的に深掘り
  - 知識データとの矛盾があれば明確に確認

  知識データがない場合:
  - 通常のオープン質問でヒアリング
```

### AIの行動

```
知識データがある場合のヒアリング開始メッセージ:

「docs/knowledge/ に既存の知識データがあります。
 事前に読み込んだ情報:
 - 課題: {pain-points.md からの要約}
 - ペルソナ: {personas.md からの要約}
 - 競合: {competitors.md からの要約}

 この情報を踏まえてヒアリングを進めます。
 既知の情報は確認形式で、未知の部分を重点的に質問します。」
```

## Stage 1: アイデアの核（約5分）

### 目的
ぼんやりしたアイデアの輪郭をつかむ

### 質問フロー

```
Q1-1 [必須] アイデアの概要
─────────────────────────────
「どんなものを作りたいか、思いつくままに教えてください。
 まとまっていなくて大丈夫です。」

→ 自由回答（制限なし）
→ AIはここで要約・確認を行う
```

```
Q1-2 [必須] きっかけ・動機
─────────────────────────────
「このアイデアを思いついたきっかけは何ですか？
 自分自身の体験ですか？誰かの悩みを聞いたのですか？」

→ 自由回答
→ 課題の実在性の初期判断に使用
```

```
Q1-3 [必須] 一言で言うと
─────────────────────────────
「今の話をまとめると、こういうことでしょうか？

 『[AIが要約した内容]』

 合っていますか？修正したい部分はありますか？」

→ Yes / 修正あり
→ 修正がある場合は再確認
```

```
Q1-4 [任意] 参考にしているサービス
─────────────────────────────
「似たサービスや、参考にしているものはありますか？
 『○○みたいな感じ』でも大丈夫です。」

→ サービス名、URL、または「特になし」
→ 競合分析のシードとして使用
```

## Stage 2: 課題の深掘り（約10分）

### 目的
解決する課題とターゲットユーザーを明確にする

### 質問フロー

```
Q2-1 [必須] ターゲットユーザー
─────────────────────────────
「このサービスを最も必要としているのは、
 どんな人ですか？具体的に教えてください。

 例: 中小企業の経営者、フリーランスのデザイナー、
     子育て中の30代女性 など」

→ 自由回答
→ ペルソナの基礎データ
```

```
Q2-2 [必須] 課題の特定
─────────────────────────────
「その人たちが抱えている一番大きな課題は
 何だと思いますか？

 できれば具体的なシーンで教えてください。
 例: 『毎月の経費精算に3時間かかっていて、
      本来やるべき仕事ができない』」

→ 自由回答
→ 課題の解像度を確認
```

```
Q2-3 [必須] 課題の深刻度
─────────────────────────────
「その課題はどのくらい深刻ですか？

 a) 毎日困っている（日常的な痛み）
 b) 時々困っている（定期的な痛み）
 c) たまに困る程度（軽い不便）
 d) あったら便利（Nice to have）」

→ 選択式
→ 課題の深刻度をスコアリング
```

```
Q2-4 [必須] 現在の対処法
─────────────────────────────
「今、その人たちはこの課題をどうやって
 対処していますか？

 a) 別のツール/サービスを使っている
 b) Excel/スプレッドシートで管理
 c) 手作業でやっている
 d) 外注している
 e) 我慢している/何もしていない
 f) その他」

→ 選択式 + 補足自由回答
→ 競合・代替手段の特定
```

```
Q2-5 [条件付き] 現在のツールの不満
─────────────────────────────
※ Q2-4で a) を選択した場合

「今使っているツール/サービスの
 不満点は何ですか？」

→ 自由回答
→ 差別化ポイントの発見
```

```
Q2-6 [任意] 課題の検証状況
─────────────────────────────
「この課題について、実際にターゲットの人に
 話を聞いたことはありますか？

 a) 5人以上に聞いた
 b) 1-4人に聞いた
 c) 自分自身の経験のみ
 d) まだ聞いていない」

→ 選択式
→ 検証の必要性を判断
```

### Stage 2 完了時の整理


## 回答→テンプレートマッピング

### 回答がどの資料のどの項目に反映されるか

| 質問 | 反映先テンプレート | 反映先セクション |

## 質問のコツ（AI向け指示）

### 質問の原則

1. **一度に1つだけ聞く**（複数の質問を同時にしない）
2. **具体例を必ず添える**（「例: ○○」で回答のハードルを下げる）
3. **「まとまっていなくてOK」を伝える**（完璧を求めない）
4. **各Stage後に整理・確認する**（認識のズレを防ぐ）
5. **選択肢を用意する**（自由回答が難しい場合の補助）

### 回答が曖昧な場合の深掘り

```
パターン1: 回答が抽象的
─────────────────────
回答: 「使いやすいツールにしたい」
深掘り: 「『使いやすい』を具体的にすると、
        例えば操作が3ステップ以内とか、
        スマホで片手で使えるとか、
        どんなイメージですか？」

パターン2: 回答が「全部」
─────────────────────
回答: 「全部の機能が必要です」
深掘り: 「もし1つだけ機能を残すとしたら、
        どれを残しますか？
        それが最も重要な機能です。」

パターン3: 回答が「わからない」
─────────────────────
回答: 「競合はわかりません」
対応: 「大丈夫です。こちらで調査しますので、
       次に進みましょう。」
→ 競合分析は後でAIが補完

パターン4: 回答が矛盾
─────────────────────
回答: 「無料で提供したいが月商100万」
対応: 「無料プランで集客して、
       有料プランで収益化するフリーミアムは
       いかがですか？」
→ 矛盾を解消する提案をする
```

## Plan Mode 80/20（設計80% / 実装20%）

### 基本原則

```
設計が完了するまで、1行もコードを書かない。

比率の目安:
  設計（Step 0〜3）: 全工数の80%
  実装（Step 4〜）:  全工数の20%

  設計が十分なら、実装は「SSOTをコードに変換するだけ」になる。
  設計が不十分だと、実装中に仕様の穴が見つかり、手戻りが発生する。
```

### 実装禁止ゲート

```
以下の条件を全て満たすまで、実装（コーディング）を開始してはならない:

Gate 1: ディスカバリー完了
────────────────────────
  □ Stage 1〜5 全て完了
  □ 全体サマリーをユーザーが承認済み

Gate 2: 事業設計完了
────────────────────────
  □ IDEA_CANVAS.md が80%以上
  □ USER_PERSONA.md が50%以上
  □ VALUE_PROPOSITION.md が50%以上

Gate 3: プロダクト設計完了
────────────────────────
  □ SSOT-0_PRD.md が完成（必須項目全て）
  □ SSOT-1_FEATURE_CATALOG.md が完成（P0機能確定）
  □ P0機能のSSOT全て監査95点以上

Gate 4: 技術設計完了
────────────────────────
  □ TECH_STACK.md が確定
  □ SSOT-3_API_CONTRACT.md が完成
  □ SSOT-4_DATA_MODEL.md が完成
  □ SSOT-5_CROSS_CUTTING.md が完成
  □ コーディング規約が確定

全ゲート通過 → 実装開始を許可
```

### AIへの指示

```
以下の状況で実装を求められた場合の対応:

ユーザー: 「とりあえずコード書いて」
AI: 「まだ設計が完了していません。
     以下のゲートが未通過です:
     □ [未完了のゲート]

     設計が80%の状態で実装に入ると、
     手戻りで結果的に時間がかかります。

     まず [次に必要なドキュメント] を完成させましょう。」

ただし、以下は例外として許可:
  - プロトタイプ/PoC（技術検証目的のみ）
  - ユーザーが明示的に「設計前だが検証目的で」と指定
```
