# Build in Public（BIP）戦略

> 開発プロセスを公開し、コミュニティと共に成長する手法

---

## 概要と原則

### Build in Publicとは

プロダクトの開発過程を公開し、フォロワーと共に成長していく手法。
透明性が信頼を生み、コミュニティがマーケティング資産になる。

### 4つの原則

1. **透明性**
   - 成功も失敗も共有
   - 数字を公開

2. **一貫性**
   - 定期的な発信
   - 長期的なコミットメント

3. **双方向性**
   - フィードバックを求める
   - コミュニティの意見を反映

4. **authenticity**
   - 飾らない
   - 本音で語る

### 透明性レベル

| レベル | 公開内容 | リスク |
|--------|---------|--------|
| Level 1 | 進捗報告、学び | 低 |
| Level 2 | 収益、ユーザー数 | 中 |
| Level 3 | 戦略、意思決定プロセス | 中 |
| Level 4 | 失敗、困難 | 高 |
| Level 5 | 財務詳細、給与 | 高 |

## 適用条件

### 向いているプロダクト

- スタートアップ
- インディーハッカー
- SaaS
- 開発者向けプロダクト

### 向いているフェーズ

- アイデア検証期
- 開発期
- 成長期（継続）

### プロジェクトタイプ別

| タイプ | 適合度 | 備考 |
|--------|-------|------|
| app | ◎ | 開発の全工程で活用 |
| lp | △ | LP公開までの過程 |
| hp | △ | リニューアル過程 |
| api | ◎ | 技術的なBIPとして最適 |
| cli | ◎ | OSS的なアプローチ |

| エンゲージメント率 | 3%以上 |

## スケジュールテンプレート

### 週次サイクル

```
月曜: 週の目標共有
火-木: 開発ログ（毎日または隔日）
金曜: 週次振り返り、学びの共有
土日: 休み or 軽い投稿
```

### 月次サイクル

```
月初: 月間目標、先月の振り返り
月中: 数字公開（収益、ユーザー数）
月末: 月間まとめ、次月の計画
```
