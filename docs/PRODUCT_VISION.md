## 🏢 ターゲット市場

### プライマリ
- **弱電・電気工事会社**（10〜100名規模）
- **設備工事・メンテナンス会社**
- **建設・リフォーム会社**

### セカンダリ
- 物流・配送業
- 清掃・ビルメンテナンス
- 介護・訪問サービス

### 共通の特徴
- 「現場」に人が出向く業態
- 事務所と現場の情報共有が課題
- デジタル化が遅れている（ホワイトボード・Excel文化）
- 中小規模で専任IT担当がいない

## 📦 プロダクト体系

### ブランド構造

```
ミエルボード（MielBoard）
└── ミエルボード for 現場
    ├── 現場WEEK（週間スケジュール）
    ├── 現場STOCK（在庫・資材管理）
    ├── 現場DRIVE（車両・アルコールチェック）
    ├── 現場LOG（日報・実績）※将来
    └── 現場AI（AIコンシェルジュ）※全モジュール横断
```

### 各プロダクトの定義

| プロダクト | 解決する問い | 主なペイン | 開発優先度 |
|-----------|-------------|-----------|-----------|
| **WEEK** | 誰が・どこで・何をしているか | ホワイトボード、朝礼、予定共有 | 🔴 Phase 1 |
| **STOCK** | 何が・どこに・いくつあるか | 在庫切れ、棚卸し、発注漏れ | 🟡 Phase 2 |
| **DRIVE** | 誰が・どう動いたか | アルコールチェック義務化、車両管理 | 🟢 Phase 3 |
| **LOG** | 何を・どれだけやったか | 日報、工数管理、原価計算 | ⚪ Phase 4 |
| **AI** | 自然言語で全てを操作 | 情報検索、レポート作成 | 🔵 全Phase並行 |

## 🤖 AIコンシェルジュ構想

### コンセプト
> **「聞けばわかる、言えば動く」**

現場の情報を自然言語で検索・操作できるAIアシスタント。
全モジュールのデータを横断的に扱える。

### ユースケース例

```
👤 「田中さん今週の予定は？」
🤖 → WEEKから取得して回答

👤 「◯◯ホテル現場に足りない資材は？」
🤖 → STOCKから在庫チェックして回答

👤 「今日アルコールチェックしてない人は？」
🤖 → DRIVEから未チェック者を抽出

👤 「来週の◯◯現場の配車を最適化して」
🤖 → WEEK + DRIVE を分析して提案

👤 「先月の工事部の稼働時間をまとめて」
🤖 → WEEK + LOG から集計してレポート生成
```

### 技術アプローチ
- LLM（GPT-4 / Claude）による自然言語理解
- Function Calling で各モジュールAPIを呼び出し
- RAG で社内データ（現場名、社員名等）を補完
- 段階的に機能追加（Phase 1: 検索のみ → Phase 2: 操作も）

## 🏗️ アーキテクチャ原則

### 1. 統一アカウント基盤

```
全プロダクトで共通:
- Organization（テナント）
- User（社員）
- Department（部門）
- Device（サイネージ）
- AuditLog（監査ログ）
```

### 2. 共通エンティティ

```
Site（現場）
- 全モジュールの"場所"の軸
- WEEK: この現場の予定
- STOCK: この現場の在庫
- DRIVE: この現場への配車
```

### 3. モジュール間連携

```
WEEK ←→ STOCK:
  予定に紐づく現場の在庫状況を表示

WEEK ←→ DRIVE:
  出発予定に対してアルコールチェック状況を表示

STOCK ←→ DRIVE:
  資材配送の車両配車を連携
```

### 4. 拡張可能なスキーマ設計

- 各モジュール固有のテーブルは独立
- 共通エンティティ（Site, User等）を軸に連携
- metadata（Json）フィールドで柔軟に拡張

## 📐 設計原則

### DRY（Don't Repeat Yourself）
- 共通機能は基盤レイヤーに集約
- モジュール間で重複コードを作らない

### 疎結合
- 各モジュールは独立してデプロイ可能
- 共通基盤への依存は明示的に

### スキーマ変更の慎重さ
- スキーマ変更は管理AI承認必須
- 拡張は metadata（Json）を優先
- 破壊的変更は避ける

### AIファースト
- データ設計時にAI活用を考慮
- 自然言語で扱いやすい命名
- 構造化されたメタデータ
